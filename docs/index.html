User Interface Module User_Interface

This is the main node and the first one spawned. This script provides the user with a small graphic explaining how to select the movement modes of the robot. It also manages the user input by going to modify, if the command is correct, the ros parameters that allow the activation of the nodes set for each mode.

rospy.set_param('mode', mode_number)

This mechanism is managed with a switch-case that makes use of standard Python dictionaries. The possible commands are the following:

    • mode1 , Reach autonomousely a user’s desired position

    • mode2 , Free drive the robot with the keyboard

    • mode3 , Drive the robot with the keyboard with an active collision avoidance

    • STOP the simulation

When the first modality is called the user will be able to set the desired goal position and to cancel it during the reaching by pressing 0 .

scripts.User_Interface.UI()[source]

This Function receives inputs from the user and manage his request by choosing the right modality thanks to a switch-case structure that makes use of python dictionaries. This logic is implemented in the module script. The user will be also able to cancel the goal request of the modality one by pressing the number 0 on his keyboard, this is simply managed changing a global bolean variable “goal_cancel”

  • No Arguments

  • No Returns

The main function prints some istruction for the user and start an infinite loop that keep listening to the user requests.

Mode 1 Module mode1

The Desired goal position is set by the user in the UI by modifying two ROS_Parameters.

  • position_x:

    rospy.set_param('des_pos_x', desired_x_position)

  • position_y:

    rospy.set_param('des_pos_y', desired_y_position)

The mode1 node sends the goal-position to the ActionServer /move_base, the client receives the feedbacks which is print on screen in the current xterm window. Moreover the action sends the the status which is: managed using python dictionaries and print, the main status are [1] the goal achievement and [2] the goal cancelation. Then the move_base node evaluate the shortest path via the Dijkstra’s path algorithm. The robot starts moving toward the goal beacause the move_base node publishes the right velocity and orientation on the cmd_vel topic.

If the user decide to change modality from the UI, the goal is correctly cancelled by sending the proper cancelation message to the ActionServer and this Node turns to an Idle state.

ROS Parameters:

  • “mode”: (type: int) parameter for setting the desired modality

  • “des_pos_x”: (type: float) parameter for setting the x desired position

  • “des_pos_y”: (type: float) parameter for setting the y desired position

Mode 3 Module mode3

The Driver Assistant Node starts from the Free Drive Node but is able to disable the commands that, if pressed, would cause the robot to collide into the walls of the map.

The robot is equipped with a laser scanner. The scanner publishes a message of type sensor_msgs/LaserScan.msg on the /scan which is read in a CallBack function by the Node Driver Assistant. Here the algorithm checks weather there’s an obstacle or not. The user will be able to select only the direction that wouldn’t bring the robot to the walls, this part has been implemented by popping out from a python dictionary the defective commands.

scripts.mode_3.clbk_laser(msg)[source]

CallBack Function to get laser scan information about the obstacles that the robot will detect in front of it. Thanks to the Subscription to the topic /scan this function can get an array of 721 elements which represent the distances of walls that the robot can see in an angle of 180 degrees. By dividing this array the robot will reecognize weather there’s an obstacle on his left, right or front.

Argument:

  • msg: (type: sensor_msgs/LaserScan.msg) array of 721 elements from the Laser Scanner

No returns
